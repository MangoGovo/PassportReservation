# 校园通行码预约管理系统 PRD

## 1. 文档信息

| 项目 | 内容 |
| --- | --- |
| 产品名称 | 校园通行码预约管理系统 |
| 适用场景 | 高校校外人员社会公众预约、公务来访预约、后台审核与统计管理 |
| 目标用户 | 游客、手机端注册用户、社会公众、公务来访人员、学校管理员、部门管理员、系统管理员、审计管理员 |
| 版本 | V1.0 |
| 日期 | 2026-06-08 |

## 2. 项目背景

疫情防控政策调整后，高校逐步开放校园。校外人员进校既包括社会公众预约，也包括因公务到访的预约。现需要建设一套校园通行码预约管理系统，支持手机端预约申请、预约查询、通行码展示，以及后台管理端的部门、管理员、预约审核、统计和安全审计。

系统需要满足等保三级相关设计要求，并对敏感数据进行加密、摘要和脱敏保护：密码使用 bcrypt 加盐哈希存储，身份证号、手机号等敏感信息加密存储，同时使用 SHA-256 保存不可逆查询摘要，页面展示时进行脱敏处理。

## 3. 产品目标

1. 支持社会公众通过预约二维码进入手机端预约页面，提交预约后自动审核通过并生成通行码。
2. 支持公务来访预约，预约提交后由对应访问部门或有权限管理员审核，通过后生成有效通行码。
3. 支持预约人通过姓名、身份证号、手机号查询本人历史预约记录并查看通行码。
4. 支持手机端游客状态识别，游客点击预约、我的预约、查看通行码等业务功能时提示登录或注册。
5. 支持学校管理员进行管理员、部门、社会公众预约、公务预约的管理、查询和统计。
6. 支持部门管理员在权限范围内审核和管理本部门公务预约。
7. 支持系统管理员配置用户、角色和权限。
8. 支持审计管理员查看系统操作日志。
9. 对登录、查看预约详情、审核、授权、统计查询等关键操作留痕，满足不少于 6 个月的日志保存要求。

| 类别 | 选型 | 说明 |

| 鉴权会话 | Sa-Token | 负责后台管理员和手机端注册用户的登录认证、Token 签发、会话管理、权限校验、登录超时控制 |

## 6. 核心业务流程

### 6.1 社会公众预约流程

1. 用户扫描预约通道二维码，进入手机端预约首页。
2. 游客状态点击“我要预约”时，系统弹出登录/注册提示；用户登录或注册成功后回到预约申请页面。
3. 用户选择预约类型“社会公众预约”。
4. 用户填写预约校区、预约进校时间、所在单位、姓名、身份证号、手机号、交通方式、车牌号、随行人员等信息。
5. 用户提交申请。
6. 系统校验必填项、身份证号、手机号、预约时间、随行人员信息。
7. 校验通过后，系统生成预约记录，审核状态为“自动通过”。
8. 系统生成通行码。
9. 当前日期为预约访问日期时，展示紫色有效通行码；不在预约访问日期当天时，展示灰色无效通行码。

### 6.2 公务预约流程

1. 用户扫描预约通道二维码，进入手机端预约首页。
2. 游客状态点击“我要预约”时，系统弹出登录/注册提示；用户登录或注册成功后回到预约申请页面。
3. 用户选择预约类型“公务预约”。
4. 用户填写基础预约信息。
5. 用户额外填写公务访问部门、公务访问接待人、来访事由。
6. 用户提交申请。
7. 系统校验字段后生成预约记录，审核状态为“待审核”。
8. 对应部门管理员或有权限管理员在后台查看待审核记录。
9. 管理员审核通过后，预约状态变为“审核通过”，系统生成通行码。
10. 管理员审核拒绝后，预约状态变为“审核拒绝”，用户查看时展示拒绝状态和拒绝原因，不展示有效通行码。

### 6.3 我的预约查询流程

1. 用户进入“我的预约”。
2. 游客状态点击“我的预约”时，系统弹出登录/注册提示；用户登录或注册成功后回到“我的预约”页面。
3. 输入姓名、身份证号、手机号。
4. 系统使用原始输入与加密存储的查询索引进行匹配，并校验当前登录用户是否有权查看该查询结果。
5. 匹配成功后展示历史预约列表，包括申请日期、进校日期、预约校区、预约类型、审核状态。
6. 用户点击“查看”，系统展示预约详情和通行码；游客或登录过期状态下需重新登录。

### 6.4 后台登录流程

1. 管理员输入登录名和密码。
2. 系统校验账号状态、锁定状态、密码有效期、密码复杂度策略。
3. 密码校验通过后调用 Sa-Token 完成登录，生成后台访问 Token。
4. 登录失败累计 5 次，账号锁定 30 分钟；失败计数和锁定 TTL 优先写入 Redis，账号最终状态同步至 MySQL。
5. 登录成功后 30 分钟无操作自动退出，由 Sa-Token 活跃超时策略控制。
6. 登录成功、失败、退出、锁定均写入审计日志。

手机端通过预约通道二维码访问，二维码可部署在校门口、学校官网、公众号或公告页。

#### 页面元素

| 元素 | 说明 |
| --- | --- |
| 系统名称 | 校园通行码预约管理系统 |

| 预约说明 | 展示进校规则、有效期说明、隐私保护说明 |

### 7.2 游客登录/注册提示

#### 触发场景

| 场景 | 处理规则 |
| --- | --- |
| 游客点击“我要预约” | 弹出登录/注册提示 |
| 游客点击“我的预约” | 弹出登录/注册提示 |
| 游客访问通行码详情链接 | 弹出登录/注册提示 |
| 登录态过期后提交表单 | 阻止提交，提示重新登录 |

#### 提示内容

提示文案建议为：“请先登录或注册后继续办理校园通行码预约。”

#### 操作规则

1. 提示框提供“登录”“注册”“取消”三个操作。
2. 点击“登录”进入手机端登录页，登录成功后返回原目标页面。
3. 点击“注册”进入手机端注册页，注册成功并登录后返回原目标页面。
4. 点击“取消”关闭提示框，停留在当前页面。
5. 原目标页面可通过 Redis 短期保存或通过前端路由参数保存，保存时间建议不超过 10 分钟。

#### 手机端注册字段

| 字段 | 必填 | 规则 |
| --- | --- | --- |
| 手机号 | 是 | 中国大陆手机号格式校验，唯一 |
| 密码 | 是 | 长度不少于 8 位，包含数字、大小写字母、特殊字符 |
| 确认密码 | 是 | 必须与密码一致 |
| 姓名 | 否 | 可注册时填写，也可首次预约时填写 |
| 身份证号 | 否 | 可注册时填写，也可首次预约时填写 |

#### 手机端登录字段

| 字段 | 必填 | 规则 |
| --- | --- | --- |
| 手机号 | 是 | 已注册手机号 |
| 密码 | 是 | 与注册密码匹配 |

#### 安全规则

1. 手机端用户密码使用 bcrypt 加盐哈希存储。
2. 手机号、身份证号存储时加密，保存 SHA-256 查询摘要。
3. 手机端登录态使用 Sa-Token 管理，Token 和会话信息存入 Redis。
4. 手机端用户连续登录失败 5 次锁定 30 分钟。

### 7.3 我要预约

#### 表单字段

| 字段 | 类型 | 必填 | 规则 |
| --- | --- | --- | --- |
| 预约类型 | 单选 | 是 | 社会公众预约、公务预约 |
| 预约校区 | 下拉 | 是 | 由后台维护校区字典 |
| 预约进校时间 | 日期时间 | 是 | 仅允许预约申请当天起 7 个自然日内的访问，含申请当天；申请日 D 可预约 D 至 D+6 |
| 所在单位 | 文本 | 是 | 2-100 字 |
| 姓名 | 文本 | 是 | 2-30 字 |
| 身份证号 | 文本 | 是 | 中国居民身份证 18 位格式校验 |
| 手机号 | 文本 | 是 | 中国大陆手机号格式校验 |
| 交通方式 | 下拉 | 是 | 步行、骑行、公共交通、自驾、其他 |
| 车牌号 | 文本 | 否 | 交通方式为自驾时建议必填；支持新能源车牌 |
| 随行人员 | 多组 | 否 | 每人包含姓名、身份证号、手机号 |
| 公务访问部门 | 下拉 | 公务必填 | 来源于部门表 |
| 公务访问接待人 | 文本 | 公务必填 | 2-30 字 |
| 来访事由 | 多行文本 | 公务必填 | 5-500 字 |

#### 提交规则

1. 社会公众预约提交后自动审核通过。
2. 公务预约提交后进入待审核状态。
3. 同一身份证号在同一校区、同一预约日期不得重复提交有效预约。
4. 预约访问日期必须在申请日期起 7 个自然日内，含申请当天；超出范围阻止提交。
5. 通行码业务周期为 1 天，仅在预约访问日期当天有效。
6. 随行人员计入预约人次。
7. 必填字段缺失、格式错误、预约时间无效时阻止提交并提示错误。
8. 身份证号、手机号入库存储前必须加密。
9. 页面展示身份证号、手机号时必须脱敏。

#### 提交成功反馈

| 预约类型 | 成功提示 |
| --- | --- |
| 社会公众预约 | 预约成功，已自动审核通过，可查看通行码 |
| 公务预约 | 预约已提交，请等待访问部门审核 |

### 7.5 通行码展示

#### 展示规则

| 条件 | 展示结果 |
| --- | --- |
| 社会公众预约审核通过且当前日期等于预约访问日期 | 紫色有效通行码 |
| 公务预约审核通过且当前日期等于预约访问日期 | 紫色有效通行码 |
| 当前日期早于预约访问日期 | 灰色无效通行码，提示“未到预约日期” |
| 当前日期晚于预约访问日期 | 灰色无效通行码，提示“预约已过期” |
| 公务预约待审核 | 不展示有效通行码，提示“待审核” |
| 公务预约审核拒绝 | 不展示有效通行码，提示“审核未通过” |
| 游客或登录态过期 | 不展示通行码，提示登录或注册 |

#### 通行码内容

二维码建议使用 zxing.jar 生成，二维码载荷内容建议为 JSON 字符串。

```json
{
  "reservationNo": "R202606080001",
  "name": "张*三",
  "idCard": "310***********1234",
  "campus": "主校区",
  "visitTime": "2026-06-08 09:00:00",
  "generatedAt": "2026-06-08 08:30:00",
  "status": "VALID"
}
```

#### 姓名脱敏规则

| 姓名长度 | 展示规则 | 示例 |
| --- | --- | --- |
| 2 字 | 中间加 `*` | 张三 -> 张*三 |
| 3 字 | 中间 1 字替换为 `*` | 张小三 -> 张*三 |
| 超过 3 字 | 中间全部替换为 `*` | 欧阳小明 -> 欧**明 |

#### 身份证号脱敏规则

1. 页面展示：保留前 3 位和后 4 位，中间用 `*` 替换。
2. 二维码展示：不得包含完整身份证号。
3. 数据库存储：身份证号密文存储，同时保存不可逆查询摘要用于等值查询。

### 7.6 后台登录

#### 功能说明

后台管理端仅面向学校内部管理人员使用，需要通过 Sa-Token 登录认证后访问。

#### 规则

1. 密码长度必须大于等于 8 位。
2. 密码必须包含数字、大写字母、小写字母、特殊字符。
3. 密码使用 bcrypt 加盐哈希后保存，不保存明文密码。
4. 密码超过 90 天未修改，登录后强制进入修改密码页面。
5. 连续登录失败 5 次锁定 30 分钟，失败计数和锁定 TTL 存入 Redis。
6. 登录后 30 分钟无操作自动退出，Sa-Token 会话和 Token 信息存入 Redis。
7. 登录、失败、退出、锁定、修改密码均记录审计日志。

### 7.7 管理员管理

#### 使用角色

学校管理员、系统管理员。

#### 功能

| 功能 | 说明 |
| --- | --- |
| 新增管理员 | 创建后台账号 |

## 8. 状态设计

### 8.1 预约类型

| 编码 | 名称 |
| --- | --- |
| PUBLIC | 社会公众预约 |
| OFFICIAL | 公务预约 |

### 8.2 审核状态

| 编码 | 名称 | 说明 |
| --- | --- | --- |
| AUTO_APPROVED | 自动通过 | 社会公众预约提交后自动进入 |
| PENDING | 待审核 | 公务预约提交后默认状态 |
| APPROVED | 审核通过 | 公务预约人工审核通过 |
| REJECTED | 审核拒绝 | 公务预约人工审核拒绝 |

### 8.3 通行码状态

| 编码 | 名称 | 说明 |
| --- | --- | --- |
| VALID | 有效 | 审核通过且当前时间在预约访问日期当天 00:00:00 至 23:59:59 内 |
| NOT_STARTED | 未生效 | 当前日期早于预约访问日期 |
| EXPIRED | 已过期 | 当前日期晚于预约访问日期 |
| UNAPPROVED | 未通过 | 待审核或审核拒绝 |

### 8.4 有效期规则

1. 预约窗口：用户仅可预约申请日期起 7 个自然日内的访问，含申请当天；申请日 D 可预约 D 至 D+6 的访问。
2. 通行码业务周期：1 天。
3. 通行码有效期：预约访问日期当天 00:00:00 至 23:59:59。
4. `valid_start_time` 按预约访问日期 00:00:00 写入，`valid_end_time` 按预约访问日期 23:59:59 写入。
5. 当前时间落在 `valid_start_time` 与 `valid_end_time` 之间且预约审核通过时，通行码显示为紫色有效；否则显示为灰色无效或对应审核状态。

## 10. 安全与隐私要求

### 10.1 身份鉴别

1. 后台账号和手机端注册用户均通过 Sa-Token 登录认证后访问受保护功能。
2. 后台账号和手机端注册用户的密码复杂度均要求长度不少于 8 位，包含数字、大写字母、小写字母、特殊字符。
3. 密码采用 bcrypt 加盐哈希存储。
4. 后台账号密码超过 90 天强制修改；手机端注册用户不强制 90 天过期，但支持主动修改密码。
5. 连续登录失败 5 次锁定 30 分钟，失败计数和锁定 TTL 存入 Redis。
6. 后台登录会话 30 分钟无操作自动退出；手机端登录态可配置较长有效期，但访问预约、通行码等敏感功能时必须校验登录态。

### 10.2 访问控制

1. 基于 Sa-Token + RBAC 的访问控制模型。
2. 后台菜单、按钮、接口均需进行权限校验，接口层通过 Sa-Token 拦截器或注解校验登录态和权限码。
3. 手机端预约提交、我的预约查询、通行码查看均需校验登录态；游客只允许访问预约首页和预约说明。
4. 部门管理员的数据范围默认限定为本部门。
5. 审计管理员只允许访问日志相关页面和接口。
6. 系统管理员负责角色和权限配置，不直接查看预约业务数据。

### 10.3 数据保密性

1. 身份证号、手机号入库存储必须加密。
2. 推荐方案：对称加密保存密文，SHA-256 保存查询摘要。
3. 页面、列表、二维码内容默认只展示脱敏信息。
4. 密钥不得硬编码在代码仓库中，应通过配置文件或环境变量加载。

### 10.4 数据完整性

1. 关键业务记录应包含创建时间、更新时间、操作人。
2. 预约记录、审核记录、审计日志不得物理删除。
3. 选做：审计日志使用 HMAC-SHA256 保存完整性校验值。

### 10.5 安全审计

1. 所有后台登录、手机端登录/注册、预约详情查看、审核、权限变更均写入审计日志。
2. 日志保留不少于 6 个月。
3. 审计日志查询应支持时间范围、操作用户、操作类型、操作结果筛选。

## 15. 验收标准

### 15.1 手机端验收

1. 用户可通过预约入口提交社会公众预约。
2. 社会公众预约提交后自动通过，并可查看通行码。
3. 用户可提交公务预约，提交后状态为待审核。
4. 公务预约审核通过前不展示有效通行码。
5. 用户可通过姓名、身份证号、手机号查询本人历史预约。
6. 游客点击“我要预约”“我的预约”或通行码详情时，系统提示登录或注册。
7. 用户登录或注册成功后，系统返回原目标页面。
8. 用户只能预约申请日期起 7 个自然日内的访问，含申请当天；超出范围时系统阻止提交。
9. 通行码仅在预约访问日期当天显示紫色有效，非预约访问日期当天显示灰色无效。
10. 二维码内容不包含完整身份证号和手机号。

### 15.2 后台验收

1. 管理员可登录后台，密码错误 5 次账号锁定 30 分钟。
2. 学校管理员可管理部门和部门管理员。
3. 学校管理员可查询、查看、统计社会公众预约。
4. 学校管理员可查询、审核、统计全部公务预约。
5. 部门管理员默认只能处理本部门公务预约。
6. 审计管理员只能查看审计日志，不能查看预约管理页面。
7. 关键操作均写入审计日志。

### 15.3 安全验收

1. 数据库不保存后台密码明文。
2. 数据库不保存身份证号、手机号明文。
3. 页面展示身份证号、手机号时完成脱敏。
4. 后台接口均校验登录态和权限。
5. 日志保存时间配置不少于 6 个月。

## 18. 风险与待确认项

| 风险/问题 | 建议处理 |
| --- | --- |
| 密码与摘要算法库选择 | Java 项目建议使用 BCryptPasswordEncoder 或 jBCrypt 处理密码，使用 JDK MessageDigest 或成熟安全库处理 SHA-256 摘要 |
| 通行码有效期口径 | 已确认通行码业务周期为 1 天，仅在预约访问日期当天有效 |
| 预约窗口口径 | 已确认当天可预约 7 个自然日内的访问，含申请当天 |
| 是否允许重复预约 | V1.0 建议同身份证号同校区同日只允许一条有效预约 |
| 是否展示完整敏感信息 | V1.0 建议后台也默认脱敏，确需明文需二次授权和审计 |
| 部门管理员授权范围 | 需要系统管理员或学校管理员在权限模块中配置 |
